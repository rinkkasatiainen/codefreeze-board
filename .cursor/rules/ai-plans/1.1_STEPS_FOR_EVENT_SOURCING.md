This document provides a comprehensive, step-by-step plan for implementing Event Sourcing with AWS, combining the initial architecture plan with the integrated automatic event publishing mechanism.

## Key Design Principles

1. **Automatic Event Publishing**: Events appended to EventStore are automatically published to EventBridge by a background Lambda
2. **At-Least-Once Guarantee**: Every event will be published at least once (may be published multiple times in error cases)
3. **Separation of Concerns**: Business capabilities only handle event storage; infrastructure handles publishing
4. **Optimistic Locking**: Version validation prevents concurrent write conflicts
5. **Idempotency**: Event recipients must handle duplicate events gracefully

## Components

### 1. EventStore (Aurora PostgreSQL Serverless v2)

**Purpose**: Append-only event storage optimized for fast reads

**Schema**:
```sql
CREATE TABLE events (
    stream_id VARCHAR(255) NOT NULL,
    event_number BIGINT NOT NULL,
    event_type VARCHAR(255) NOT NULL,
    event_data JSONB NOT NULL,
    event_metadata JSONB,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    published BOOLEAN NOT NULL DEFAULT FALSE,
    published_date TIMESTAMP WITH TIME ZONE,
    PRIMARY KEY (stream_id, event_number)
);

CREATE INDEX idx_stream_id ON events(stream_id);
CREATE INDEX idx_stream_event_number ON events(stream_id, event_number);
CREATE INDEX idx_unpublished_events ON events(published, created_at) WHERE published = FALSE;
```

**Key Features**:
- Composite primary key: `(stream_id, event_number)`
- `event_number` is managed by the application layer (not auto-increment)
- `published` flag tracks EventBridge publishing status
- `published_date` records when event was published
- Optimized indexes for reading events by stream
- Partial index on unpublished events for efficient background processing
- JSONB for flexible event data storage

### 2. Event Bus (Amazon EventBridge)

**Purpose**: Publish all events for business capability projection listeners

**Structure**:
- Custom event bus: `codefreeze-event-bus`
- Event pattern: All events from EventStore
- Event format:
  ```json
  {
    "source": "event-store",
    "detail-type": "EventStored",
    "detail": {
      "streamId": "stream-123",
      "eventNumber": 1,
      "eventType": "UserCreated",
      "eventData": {...},
      "eventMetadata": {...}
    }
  }
  ```

**How Business Capabilities Listen to Events**:
- Business capabilities create **EventBridge Rules** that filter events and target their Lambda functions
- Each business capability creates its own EventBridge Rule with:
  - Event pattern (e.g., filter by `eventType`, `source`, `detail-type`)
  - Target: Lambda function (direct invocation)
- When an event matches a rule, EventBridge **directly invokes** the Lambda function
- No SQS/SNS required (though they can be used if needed for additional buffering/retry logic)
- Multiple business capabilities can listen to the same events by creating multiple rules

**Example EventBridge Rule** (created by business capability):
```typescript
// In business capability's CDK construct
new events.Rule(this, 'SessionProjectionRule', {
  eventBus: eventSourcing.eventBus,
  eventPattern: {
    source: ['event-store'],
    detailType: ['EventStored'],
    detail: {
      eventType: ['SessionCreated', 'SessionUpdated', 'SessionDeleted']
    }
  },
  targets: [new targets.LambdaFunction(sessionProjectionLambda)]
});
```

**Note**: Business capabilities create their own Lambda functions and EventBridge Rules that listen to events and build their own projections. Each business capability can have different projection structures (e.g., scheduling has different read model than session management).

### 3. Event Publisher Lambda (Background Process)

**Purpose**: Automatically publish events from EventStore to EventBridge

**Key Features**:
- Scheduled execution (every 30 seconds for MVP)
- Queries unpublished events (`published = FALSE`)
- Publishes to EventBridge
- Updates `published = TRUE` and `published_date` in transaction
- Handles errors and retries automatically
- Processes events in batches (100 events per invocation)

**Location**: Infrastructure code in `cdk/event-publisher-lambda/`

### 4. Library Functions (cfb-event-sourcing package)

**Purpose**: Provide reusable library functions for event sourcing operations that can be used by business capabilities in their Lambda functions.

**Location**: `cfb-event-sourcing/src/`

**Design Principles**:
1. **Factory pattern** - Encapsulate configuration. Create instance once at module level.
2. **Eventually monad** - All async methods return `Eventually` from `@rinkkasatiainen/cfb-functional`. This makes error handling explicit: use `fold()` to handle success/failure, `bind()` for composition. No try-catch needed.

**Dependencies**: `@rinkkasatiainen/cfb-functional` (for Eventually monad)

#### 4.1 Event Store Factory
**File**: `cfb-event-sourcing/src/event-store.js`

**Factory Function**: `createEventStore(options?)`
- Creates an EventStore instance
- Reads configuration from environment variables if options not provided
- Environment variables: `EVENT_STORE_CLUSTER_ARN`, `EVENT_STORE_SECRET_ARN`, `EVENT_STORE_DB_NAME`, `AWS_REGION`
- Options can override any of these for testing

**Instance Methods** (all return `Promise<Eventually>`):
- `eventStore.getStreamVersion(streamId)` → `Eventually.resolve(number)` or `Eventually.reject(error)`
- `eventStore.appendEvent(streamId, eventType, eventData, expectedVersion, eventMetadata?)` → `Eventually.resolve(event)` or `Eventually.reject(ConcurrencyError|error)`
- `eventStore.readEvents(streamId, fromEventNumber = 0)` → `Eventually.resolve(events[])` or `Eventually.reject(error)`
- `eventStore.replayEvents(streamId, fromEventNumber = 0)` → Same as `readEvents`, named for clarity

- `eventStore.getStreamVersion(streamId)`
  - Gets current version (max event_number) for a stream
  - Returns 0 if stream doesn't exist

- `eventStore.replayEvents(streamId, fromEventNumber = 0)`
  - Convenience method for projection listeners rebuilding projections
  - Same as `readEvents()` but named for clarity of intent

**Internal Factory**: `createEventStoreInternal(options?)` (for Event Publisher Lambda)
- Includes all public methods plus:
- `eventStore.getUnpublishedEvents(batchSize)` - Query for unpublished events
- `eventStore.markEventAsPublished(streamId, eventNumber)` - Update published status

#### 4.2 Event Bus Factory
**File**: `cfb-event-sourcing/src/event-bus.js`

**Factory Function**: `createEventBus(options?)` (Internal use by Event Publisher Lambda)
- Creates an EventBus instance for publishing events
- Reads configuration from environment variables if options not provided
- Environment variables: `EVENT_BUS_NAME`, `AWS_REGION`
- **Not part of public API** - business capabilities don't use this

**Instance Methods**:
- `eventBus.publishEvent(eventDetail)` - Publishes single event to EventBridge
- `eventBus.publishEventBatch(events)` - Publishes multiple events (batches of 10)

### 5. Projections (Business Capabilities)

**Purpose**: Business capabilities create their own read-optimized projections

**Key Points**:
- **Projections belong to business capabilities**, not the event sourcing framework
- Each business capability defines its own DynamoDB table(s) for projections
- Each business capability creates Lambda functions that listen to EventBridge events
- Projection structure varies by business capability (e.g., scheduling has different read model than session management)
- Business capabilities use `readEvents()` or `replayEvents()` from the library to rebuild projections if needed

**Example**: 
- Scheduling capability might have a `schedules` DynamoDB table with columns like `eventId`, `sessions`, `timeSlots`, etc.
- Session capability might have a `sessions` DynamoDB table with columns like `sessionId`, `title`, `speakers`, `room`, etc.

**Projection Listener Pattern**:
1. Business capability creates Lambda function for projection logic
2. Business capability creates EventBridge Rule that:
   - Filters events (e.g., by `eventType`, `source`, etc.)
   - Targets the Lambda function (direct invocation)
3. When EventBridge publishes an event, it matches against rules and invokes matching Lambda functions
4. Lambda receives event from EventBridge (direct invocation, no SQS/SNS)
5. Lambda updates its own DynamoDB projection table(s) based on the event
6. Each business capability implements its own projection logic and idempotency handling

**Architecture Flow**:
```
Event Publisher Lambda → EventBridge Custom Bus → EventBridge Rules → Lambda Functions (Projection Listeners)
                                                      ↓
                                              (Each business capability
                                               creates its own rule)
```

  - RDS Data API access (read unpublished events, update published status)
  - Secrets Manager read access
  - EventBridge PutEvents permission

**CDK Code**:
```typescript
// Event Publisher Lambda
const eventPublisherLambda = new lambda.Function(this, 'EventPublisherLambda', {
  runtime: lambda.Runtime.NODEJS_22_X,
  handler: 'index.eventPublisherHandler',
  code: lambda.Code.fromAsset(path.join(__dirname, '../event-publisher-lambda')),
  environment: {
    EVENT_STORE_CLUSTER_ARN: eventStoreCluster.clusterArn,
    EVENT_STORE_SECRET_ARN: eventStoreSecret.secretArn,
    EVENT_STORE_DB_NAME: 'eventstore',
    EVENT_BUS_NAME: eventBus.eventBusName,
    AWS_REGION: this.region,
    BATCH_SIZE: '100', // events to process per invocation
  },
  timeout: Duration.minutes(5),
});

// Grant permissions
eventStoreCluster.grantDataApiAccess(eventPublisherLambda);
eventStoreSecret.grantRead(eventPublisherLambda);
eventBus.grantPutEvents(eventPublisherLambda);

// Scheduled Rule (runs every 30 seconds)
new events.Rule(this, 'EventPublisherSchedule', {
  schedule: events.Schedule.rate(Duration.seconds(30)),
  targets: [new targets.LambdaFunction(eventPublisherLambda)],
});
```

**Note**: The Event Publisher Lambda code should be in a separate location (e.g., `cdk/event-publisher-lambda/`) since it's infrastructure code, not part of the `cfb-event-sourcing` library.

#### Step 2.2: Implement Event Publisher Logic
**File**: `cdk/event-publisher-lambda/src/event-publisher.js`

**Functionality**:
1. Query EventStore for events where `published = FALSE`
2. Order by `created_at` ascending (publish oldest events first)
3. For each unpublished event:
   - Publish to EventBridge
   - Update event record: set `published = TRUE`, `published_date = NOW()`
   - Use database transaction to ensure atomicity
4. Handle errors gracefully (log, retry next cycle)

**Batch Processing**:
- Process events in batches (e.g., 100 events per invocation)
- Use pagination to handle large backlogs
- Continue processing until no more unpublished events

**Error Handling**:
- If EventBridge publish fails: Log error, leave `published = FALSE`, retry in next cycle
- If database update fails after successful publish: Log error, event may be republished (idempotent in EventBridge)
- Use idempotency keys in EventBridge to prevent duplicate processing on recipient side

## At-Least-Once Guarantee

**How it works**:

1. **Normal Flow**:
   - Event appended with `published = FALSE`
   - Event Publisher Lambda runs, publishes event, sets `published = TRUE`
   - ✅ Event published exactly once

2. **EventBridge Publish Fails**:
   - Event appended with `published = FALSE`
   - Event Publisher Lambda runs, EventBridge publish fails
   - Event remains `published = FALSE`
   - Next cycle retries publishing
   - ✅ Event eventually published (at least once)

3. **Database Update Fails After Successful Publish**:
   - Event appended with `published = FALSE`
   - Event Publisher Lambda runs, EventBridge publish succeeds
   - Database update fails (network issue, etc.)
   - Event remains `published = FALSE`
   - Next cycle republishes to EventBridge
   - ✅ Event published multiple times (at-least-once guarantee)
   - Event recipients must be idempotent

4. **Lambda Crashes Mid-Processing**:
   - Event appended with `published = FALSE`
   - Event Publisher Lambda crashes after publishing but before updating DB
   - Event remains `published = FALSE`
   - Next cycle republishes
   - ✅ Event published multiple times (at-least-once guarantee)

**Idempotency in Event Recipients**:
- Event recipients (projection listener Lambdas) must handle duplicate events
- Use `(stream_id, event_number)` as unique identifier
- Each business capability implements its own idempotency logic in its projection listeners
- Check if event has already been processed before applying to projection

## Simplifications for MVP

1. **Database**: Use RDS Data API instead of connection pooling (simpler, serverless-friendly)
2. **VPC**: Use default VPC or public subnet initially (simpler setup)
3. **Migration**: Manual SQL script instead of automated migration
4. **Projection**: Business capabilities create their own projection tables and logic
5. **Error Handling**: Basic retry logic (exponential backoff, max 3 retries)
6. **Monitoring**: Basic CloudWatch logs (add metrics/alarms later)
7. **Publishing Schedule**: 30 seconds interval (good balance between latency and cost)

## Event Listening Patterns

### Default Pattern: Direct EventBridge-to-Lambda

**Recommended for MVP**: Business capabilities create EventBridge Rules that directly invoke Lambda functions.

**Flow**:
```
Event Publisher Lambda → EventBridge Bus → EventBridge Rule → Lambda Function
```

**Benefits**:
- Simple setup (just create a Rule)
- Low latency (direct invocation)
- Automatic retry on Lambda failure (EventBridge retries)
- Built-in filtering via Rule patterns
- No additional infrastructure (no SQS/SNS)

**When to Use**: Most cases, especially for MVP

### Alternative Pattern: EventBridge → SQS → Lambda

**Use Case**: When you need message buffering, dead-letter queues, or more control over retry logic.

**Flow**:
```
Event Publisher Lambda → EventBridge Bus → EventBridge Rule → SQS Queue → Lambda Function
```

**Benefits**:
- Message buffering (events queued if Lambda is slow)
- Dead-letter queue for failed messages
- More control over retry logic
- Better for high-volume scenarios

**When to Use**: High-volume scenarios, need DLQ, or want more control over retries

### Alternative Pattern: EventBridge → SNS → Lambda

**Use Case**: When you need fan-out to multiple subscribers or want to add additional processing.

**Flow**:
```
Event Publisher Lambda → EventBridge Bus → EventBridge Rule → SNS Topic → Lambda Functions (multiple)
```

**Benefits**:
- Fan-out to multiple Lambda functions
- Can add additional subscribers (email, SMS, etc.)
- More flexible routing

**When to Use**: Need to fan-out to multiple subscribers or add non-Lambda targets

**Note**: For MVP, use the direct EventBridge-to-Lambda pattern. Add SQS/SNS later if needed.

## Future Enhancements (Post-MVP)

1. **Real-Time Publishing** (EventBridge Pipes + RDS CDC):
   - Use Aurora Database Activity Streams or DMS CDC
   - Near-real-time publishing (no polling delay)
   - More efficient for high-throughput scenarios

2. **Connection Pooling**: RDS Proxy for better performance
3. **VPC**: Proper VPC setup with private subnets
4. **Multi-Region**: Cross-region replication
5. **Event Snapshots**: Periodic snapshots for faster replay
6. **Multiple Projections**: Support multiple projection types
7. **Event Versioning**: Schema evolution support
8. **Idempotency Keys**: Add idempotency key to EventBridge events based on `(streamId, eventNumber)`
9. **Dead Letter Queue**: Events that fail to publish after N retries
10. **Monitoring**: CloudWatch metrics, alarms, dashboards
11. **Event Archiving**: Move old events to S3/Glacier
12. **Partitioning**: Multiple Event Publisher Lambdas for high throughput

## AWS Resource Choices Rationale

1. **Aurora PostgreSQL Serverless v2**: 
   - Optimized for append-only workloads
   - Auto-scaling
   - Fast reads with indexes
   - PostgreSQL JSONB support

2. **EventBridge** (vs DynamoDB Streams):
   - Better for event sourcing patterns
   - Decoupled architecture
   - Built-in filtering and routing via Rules
   - Multiple subscribers possible (each creates its own Rule)
   - Direct Lambda invocation (no SQS/SNS required, but can be used if needed)
   - Better for cross-service events
   - Business capabilities can filter events they care about

3. **DynamoDB for Projections** (created by business capabilities):
   - Fast read/write for projections
   - Conditional updates (optimistic locking)
   - Serverless (pay per use)
   - Good for key-value projections
   - Each business capability designs its own table structure

4. **RDS Data API** (initially):
   - No connection management
   - Serverless-friendly
   - Simpler than connection pooling
   - Good for Lambda

5. **Scheduled Lambda for Publishing**:
   - Simple to implement
   - Reliable retry mechanism
   - Cost-effective for MVP
   - Can be upgraded to CDC-based solution later

## Environment Variables Needed

**For Lambda Functions using the library (Event Writers)**:
- `EVENT_STORE_CLUSTER_ARN` - RDS cluster ARN (from CDK construct)
- `EVENT_STORE_SECRET_ARN` - Secrets Manager ARN (from CDK construct)
- `EVENT_STORE_DB_NAME` - Database name (e.g., "eventstore")
- `AWS_REGION` - AWS region (usually available in Lambda runtime)

**For Lambda Functions (Projection Listeners)**:
- `EVENT_BUS_NAME` - EventBridge bus name (from CDK construct) - used when creating EventBridge Rule
- `AWS_REGION` - AWS region
- Business capabilities set their own environment variables for their DynamoDB projection tables

**Note**: 
- Business capabilities do NOT need `EVENT_BUS_NAME` for writing events - they don't publish directly
- EventBridge Rules are created in CDK and automatically grant invoke permission to Lambda
- Lambda functions receive events directly from EventBridge (no SQS/SNS in the default pattern)

**For Event Publisher Lambda**:
- `EVENT_STORE_CLUSTER_ARN` - RDS cluster ARN
- `EVENT_STORE_SECRET_ARN` - Secrets Manager ARN
- `EVENT_STORE_DB_NAME` - Database name
- `EVENT_BUS_NAME` - EventBridge bus name
- `AWS_REGION` - AWS region
- `BATCH_SIZE` - Number of events to process per invocation (default: 100)

**Usage in Business Capability Lambda (Event Writer)**:

All EventStore methods return `Eventually` monads from `@rinkkasatiainen/cfb-functional`.
Use `fold()` to handle success/failure, or chain with `bind()` for composition.

```javascript
import { createEventStore, ConcurrencyError } from '@rinkkasatiainen/cfb-event-sourcing';

// Create once at module level - reads config from environment variables
const eventStore = createEventStore();

export const handler = async (event) => {
  // Get current version (returns Eventually)
  const versionResult = await eventStore.getStreamVersion(event.streamId);
  
  // Chain with bind to append event if version lookup succeeded
  const appendResult = await versionResult.bind(currentVersion =>
    eventStore.appendEvent(
      event.streamId,
      event.eventType,
      event.eventData,
      currentVersion // expectedVersion
    )
  );
  
  // Fold to convert Eventually to HTTP response
  return appendResult.fold(
    // Error case
    error => {
      if (error instanceof ConcurrencyError) {
        return { statusCode: 409, body: JSON.stringify({ error: 'Version conflict' }) };
      }
      return { statusCode: 500, body: JSON.stringify({ error: error.message }) };
    },
    // Success case - event will be automatically published by Event Publisher Lambda
    appendedEvent => ({ statusCode: 200, body: JSON.stringify(appendedEvent) })
  );
};
```

**Usage in Business Capability Lambda (Projection Listener)**:
```javascript
import { DynamoDBClient } from '@aws-sdk/client-dynamodb';
import { DynamoDBDocumentClient, PutCommand } from '@aws-sdk/lib-dynamodb';

// This Lambda is triggered by EventBridge via an EventBridge Rule
// The EventBridge Rule filters events and directly invokes this Lambda
export const handler = async (eventBridgeEvent) => {
  // EventBridge event structure:
  // {
  //   "source": "event-store",
  //   "detail-type": "EventStored",
  //   "detail": { streamId, eventNumber, eventType, eventData, ... }
  // }
  const event = eventBridgeEvent.detail; // Event from EventBridge
  
  // Business capability's own DynamoDB table
  const dynamoClient = DynamoDBDocumentClient.from(new DynamoDBClient({}));
  const projectionTable = process.env.MY_PROJECTION_TABLE_NAME;
  
  // Business capability implements its own projection logic
  // Example: Update scheduling projection
  await dynamoClient.send(new PutCommand({
    TableName: projectionTable,
    Item: {
      eventId: event.streamId,
      eventNumber: event.eventNumber,
      eventType: event.eventType,
      // ... business capability's own projection structure
      updatedAt: new Date().toISOString()
    }
  }));
  
  // Handle idempotency: Check if event already processed using (streamId, eventNumber)
  // Each business capability implements its own idempotency logic
};
```

## Security Considerations

1. **Database Access**: 
   - Use Secrets Manager for credentials
   - Lambda functions only (no public access)
   - Security groups restrict access

2. **API Gateway**: 
   - Add authentication (Cognito) in future
   - Rate limiting
   - CORS configuration

3. **EventBridge**: 
   - Resource-based policies
   - Event source verification

4. **DynamoDB** (created by business capabilities): 
   - IAM policies for table access (managed by business capabilities)
   - Encryption at rest

## Migration Path

For existing implementations using the original plan:

1. **Deploy new infrastructure** (Event Publisher Lambda, database migration)
2. **Update library** (new version of `cfb-event-sourcing`)
3. **Update business capabilities**:
   - Remove `publishEvent()` calls
   - Update `appendEvent()` calls (remove EventBridge config)
   - Deploy updated Lambda functions
4. **Verify**: Check that events are being published automatically
5. **Monitor**: Watch for any unpublished events (should be near zero)

**Backward Compatibility**:
- Old code calling `publishEvent()` will still work (function exists but is deprecated/internal)
- Events appended with old code will be automatically published by Event Publisher Lambda
