# Dead Letter Queue for Event Handling

This document describes how to implement a dead letter queue (DLQ) pattern for events that cannot be successfully processed by Lambda handlers after retries. This is an extension to the Event Sourcing architecture described in `1.1_STEPS_FOR_EVENT_SOURCING.md`.

## Overview

**Problem**: When a projection listener Lambda function fails to process an event (throws an exception), we need:
1. Automatic retries (e.g., 10 attempts) before giving up
2. A dead letter queue to capture events that failed after all retries
3. Monitoring and alerting when events end up in the DLQ
4. Manual intervention capability to investigate and reprocess failed events

**Solution**: Use the **EventBridge → SQS → Lambda** pattern with SQS Dead Letter Queue. This provides:
- Configurable retry logic (via SQS visibility timeout and receive count)
- Automatic DLQ routing after max retries
- Better control over retry behavior
- Message persistence and durability

## Architecture

### Current Architecture (Direct EventBridge → Lambda)

```
Event Publisher Lambda → EventBridge Bus → EventBridge Rule → Lambda Function (direct invocation)
```

**Limitations**:
- EventBridge has built-in retry (2 attempts by default, configurable up to 185)
- Lambda can have a DLQ, but limited retry control
- Harder to monitor and manage retries

### Recommended Architecture (EventBridge → SQS → Lambda with DLQ)

```
Event Publisher Lambda → EventBridge Bus → EventBridge Rule → SQS Queue → Lambda Function
                                                                             ↓ (after 10 failures)
                                                                    SQS Dead Letter Queue
```

**Benefits**:
- **10 retries** via SQS message visibility timeout and receive count
- **Automatic DLQ routing** after max retries
- **Message persistence** (messages survive Lambda failures)
- **Better monitoring** (SQS metrics for queue depth, DLQ size)
- **Flexibility** (can manually republish from DLQ)

## Components

### 1. SQS Queue (Main Queue)

**Purpose**: Buffer events before Lambda processing

**Configuration**:
- **Message Visibility Timeout**: 30 seconds (adjust based on Lambda timeout)
- **Receive Message Wait Time**: 20 seconds (long polling)
- **Maximum Receives**: 10 (after 10 attempts, move to DLQ)
- **Message Retention Period**: 14 days (default)

**Why SQS?**:
- Decouples EventBridge from Lambda
- Provides message buffering
- Enables controlled retry via visibility timeout
- Automatic DLQ routing based on receive count

### 2. SQS Dead Letter Queue (DLQ)

**Purpose**: Store events that failed after 10 retry attempts

**Configuration**:
- **Message Retention Period**: 14 days (recommended for investigation)
- **No receive count limit** (messages stay until manually processed or expired)

**Use Cases**:
- Manual investigation of failed events
- Reprocessing after bug fixes
- Analytics on failure patterns

### 3. Lambda Function Configuration

**Purpose**: Process events from SQS queue

**Configuration**:
- **Trigger**: SQS queue (batch size: 1-10 messages)
- **Reserved Concurrency**: Optional (to control throughput)
- **Timeout**: Should be less than SQS visibility timeout
- **Error Handling**: Throw exception on failure (SQS will retry)

**Note**: Lambda should throw exceptions for transient failures. SQS will automatically retry based on visibility timeout and receive count.

### 4. EventBridge Rule Configuration

**Purpose**: Route events from EventBridge to SQS

**Configuration**:
- **Target**: SQS Queue (instead of Lambda directly)
- **Retry Policy**: Not used (SQS handles retries)
- **Dead Letter Queue**: Not needed (SQS DLQ handles failures)

## Implementation

### CDK Implementation

#### Step 1: Create SQS Dead Letter Queue

**File**: `cdk/lib/event-sourcing-resources.ts` (or in business capability construct)

```typescript
import * as sqs from 'aws-cdk-lib/aws-sqs';

// Dead Letter Queue
const projectionHandlerDLQ = new sqs.Queue(this, 'ProjectionHandlerDLQ', {
  queueName: 'projection-handler-dlq',
  retentionPeriod: Duration.days(14),
  visibilityTimeout: Duration.seconds(30), // Not critical for DLQ
});

// Grant permissions to send messages to DLQ (for SQS main queue)
projectionHandlerDLQ.grantSendMessages(new iam.ServicePrincipal('sqs.amazonaws.com'));
```

#### Step 2: Create SQS Main Queue with DLQ Redrive Policy

```typescript
import * as sqs from 'aws-cdk-lib/aws-sqs';

// Main Queue with DLQ
const projectionHandlerQueue = new sqs.Queue(this, 'ProjectionHandlerQueue', {
  queueName: 'projection-handler-queue',
  visibilityTimeout: Duration.seconds(30), // Should be >= Lambda timeout
  receiveMessageWaitTime: Duration.seconds(20), // Long polling
  retentionPeriod: Duration.days(14),
  deadLetterQueue: {
    queue: projectionHandlerDLQ,
    maxReceiveCount: 10, // After 10 attempts, move to DLQ
  },
});
```

**Key Configuration**:
- **`maxReceiveCount: 10`**: After 10 failed attempts, message moves to DLQ
- **`visibilityTimeout`**: Must be >= Lambda timeout + processing time
- **`receiveMessageWaitTime: 20`**: Long polling reduces API calls

#### Step 3: Configure Lambda Function with SQS Trigger

```typescript
import * as lambda from 'aws-cdk-lib/aws-lambda';
import * as sqs from 'aws-cdk-lib/aws-sqs';

// Lambda function
const projectionLambda = new lambda.Function(this, 'ProjectionLambda', {
  runtime: lambda.Runtime.NODEJS_22_X,
  handler: 'index.projectionHandler',
  code: lambda.Code.fromAsset(path.join(__dirname, '../../functions')),
  timeout: Duration.seconds(25), // Must be < visibilityTimeout (30s)
  environment: {
    PROJECTION_TABLE_NAME: projectionTable.tableName,
    AWS_REGION: this.region,
  },
});

// Add SQS event source
projectionLambda.addEventSource(
  new lambdaEventSources.SqsEventSource(projectionHandlerQueue, {
    batchSize: 1, // Process one message at a time for better error isolation
    maxBatchingWindow: Duration.seconds(5), // Optional: batch multiple messages
  })
);

// Grant Lambda permission to consume from SQS
projectionHandlerQueue.grantConsumeMessages(projectionLambda);
```

**Key Points**:
- Lambda timeout (25s) < SQS visibility timeout (30s) ensures Lambda completes before message becomes visible again
- `batchSize: 1` processes one message at a time for better error isolation
- SQS automatically retries failed messages based on visibility timeout

#### Step 4: Update EventBridge Rule to Target SQS Instead of Lambda

```typescript
import * as events from 'aws-cdk-lib/aws-events';
import * as targets from 'aws-cdk-lib/aws-events-targets';

// EventBridge Rule targets SQS Queue (not Lambda directly)
new events.Rule(this, 'ProjectionRule', {
  eventBus: eventSourcing.eventBus,
  eventPattern: {
    source: ['event-store'],
    detailType: ['EventStored'],
    detail: {
      eventType: ['SessionCreated', 'SessionUpdated'],
    },
  },
  targets: [
    new targets.SqsQueue(projectionHandlerQueue, {
      messageGroupId: undefined, // Only needed for FIFO queues
    }),
  ],
});

// Grant EventBridge permission to send to SQS
projectionHandlerQueue.grantSendMessages(new iam.ServicePrincipal('events.amazonaws.com'));
```

#### Step 5: Add CloudWatch Alarms for Monitoring

```typescript
import * as cloudwatch from 'aws-cdk-lib/aws-cloudwatch';
import * as sns from 'aws-cdk-lib/aws-sns';

// Alarm for DLQ messages

```

### Lambda Handler Implementation

**File**: `functions/src/projection-handler.js`

```javascript
import { DynamoDBClient } from '@aws-sdk/client-dynamodb';
import { DynamoDBDocumentClient, PutCommand } from '@aws-sdk/lib-dynamodb';

/**
 * Lambda handler triggered by SQS events from EventBridge
 * 
 * @param {Object} sqsEvent - SQS event containing EventBridge events
 * @param {Object} context - Lambda context
 */
export const projectionHandler = async (sqsEvent, context) => {
  const dynamoClient = DynamoDBDocumentClient.from(new DynamoDBClient({}));
  const tableName = process.env.PROJECTION_TABLE_NAME;

  // Process each SQS record (batchSize: 1 means one record per invocation)
  for (const record of sqsEvent.Records) {
    try {
      // Parse EventBridge event from SQS message body
      const eventBridgeEvent = JSON.parse(record.body);
      
      // Extract event detail (EventBridge format)
      const event = eventBridgeEvent.detail;
      const { streamId, eventNumber, eventType, eventData } = event;

      // Validate event
      if (!streamId || !eventNumber || !eventType) {
        throw new Error(`Invalid event: missing required fields. Event: ${JSON.stringify(event)}`);
      }

      // Check idempotency: Get current projection
      const currentProjection = await dynamoClient.send(new GetCommand({
        TableName: tableName,
        Key: { streamId },
      }));

      // Skip if already processed
      if (currentProjection.Item && currentProjection.Item.lastEventNumber >= eventNumber) {
        console.log(`Event ${eventNumber} for stream ${streamId} already processed`);
        return; // Success (message will be deleted from queue)
      }

      // Process event: Update projection
      await dynamoClient.send(new PutCommand({
        TableName: tableName,
        Item: {
          streamId,
          lastInput: eventData.input,
          lastEventNumber: eventNumber,
          updatedAt: eventData.timestamp || new Date().toISOString(),
        },
      }));

      console.log(`Successfully processed event ${eventNumber} for stream ${streamId}`);
      
    } catch (error) {
      console.error('Error processing event:', error);
      // Throw error to trigger SQS retry
      // After 10 retries, message will be moved to DLQ
      throw error;
    }
  }
};
```

**Key Points**:
- **Throw exceptions** on failure (SQS will retry)
- **Don't catch and swallow errors** (unless you want to skip the message)
- **Process events idempotently** (check if already processed)
- **Log errors** for debugging

## Retry Behavior

### How Retries Work

1. **EventBridge publishes event** → SQS Queue receives message
2. **Lambda triggers** from SQS queue
3. **Lambda processes message**:
   - **Success**: Message deleted from queue ✅
   - **Failure (exception thrown)**: Message becomes visible again after visibility timeout
4. **Message becomes visible** after visibility timeout (30 seconds)
5. **Lambda retries** (up to 10 times)
6. **After 10 failures**: Message moved to DLQ automatically

### Retry Timeline Example

Assuming `visibilityTimeout: 30s` and `maxReceiveCount: 10`:

```
Attempt 1: Lambda fails → Message invisible for 30s
Attempt 2: Lambda fails → Message invisible for 30s
...
Attempt 10: Lambda fails → Message invisible for 30s
Attempt 11: SQS detects maxReceiveCount reached → Move to DLQ
```

**Total retry window**: ~5 minutes (10 attempts × 30s visibility timeout)

### Adjusting Retry Behavior

**To change retry count**:
```typescript
deadLetterQueue: {
  queue: projectionHandlerDLQ,
  maxReceiveCount: 5, // Change from 10 to 5
}
```

**To change retry interval** (visibility timeout):
```typescript
const queue = new sqs.Queue(this, 'Queue', {
  visibilityTimeout: Duration.seconds(60), // Increase to 60s between retries
});
// Remember: Lambda timeout must be < visibilityTimeout
```

**To enable exponential backoff** (advanced):
Use SQS message attributes or implement custom retry logic in Lambda (not recommended - SQS handles this automatically).

## Monitoring and Alerting

### CloudWatch Metrics

**SQS Queue Metrics**:
- `ApproximateNumberOfMessagesVisible`: Messages waiting to be processed
- `ApproximateNumberOfMessagesNotVisible`: Messages being processed
- `NumberOfMessagesSent`: Events received from EventBridge
- `NumberOfMessagesReceived`: Messages delivered to Lambda
- `NumberOfMessagesDeleted`: Successfully processed messages
- `ApproximateAgeOfOldestMessage`: Age of oldest unprocessed message

**SQS DLQ Metrics**:
- `ApproximateNumberOfMessagesVisible`: Failed events in DLQ (most important)
- `ApproximateNumberOfMessagesNotVisible`: Messages being inspected/reprocessed

**Lambda Metrics**:
- `Invocations`: Number of Lambda invocations
- `Errors`: Lambda failures
- `Duration`: Processing time
- `Throttles`: When Lambda is throttled (concurrency limit)

### Recommended Alarms

1. **DLQ Message Alarm** (Critical):
   ```typescript
   new cloudwatch.Alarm(this, 'DLQAlarm', {
     metric: dlq.metricApproximateNumberOfMessagesVisible(),
     threshold: 1,
     evaluationPeriods: 1,
   });
   ```

2. **Queue Depth Alarm** (Warning):
   ```typescript
   new cloudwatch.Alarm(this, 'QueueDepthAlarm', {
     metric: queue.metricApproximateNumberOfMessagesVisible(),
     threshold: 100, // Alert if > 100 messages waiting
     evaluationPeriods: 2,
   });
   ```

3. **Oldest Message Age Alarm** (Warning):
   ```typescript
   new cloudwatch.Alarm(this, 'OldMessageAlarm', {
     metric: queue.metricApproximateAgeOfOldestMessage(),
     threshold: 300, // Alert if oldest message > 5 minutes
     evaluationPeriods: 1,
   });
   ```

## Handling DLQ Messages

### Manual Investigation

1. **View DLQ messages** in AWS Console (SQS → Dead Letter Queue)
2. **Inspect message body** (contains EventBridge event)
3. **Check CloudWatch Logs** for error details
4. **Identify root cause** (bug, data issue, dependency failure, etc.)

### Reprocessing DLQ Messages

**Option 1: Republish to Main Queue** (after fixing bug):

```javascript
import { SQSClient, SendMessageCommand } from '@aws-sdk/client-sqs';

const sqsClient = new SQSClient({});

async function reprocessDLQMessage(dlqMessage) {
  // Extract original event from DLQ message
  const eventBridgeEvent = JSON.parse(dlqMessage.body);
  
  // Send back to main queue
  await sqsClient.send(new SendMessageCommand({
    QueueUrl: process.env.MAIN_QUEUE_URL,
    MessageBody: dlqMessage.body,
  }));
  
  // Delete from DLQ after successful republish
  // (handled separately to avoid message loss)
}
```

**Option 2: Create Lambda Function to Republish DLQ Messages**:

```typescript
// CDK: Lambda function triggered by manual invocation or scheduled rule
const dlqReprocessorLambda = new lambda.Function(this, 'DLQReprocessor', {
  runtime: lambda.Runtime.NODEJS_22_X,
  handler: 'index.reprocessDLQ',
  code: lambda.Code.fromAsset(path.join(__dirname, '../../dlq-reprocessor')),
  environment: {
    DLQ_URL: projectionHandlerDLQ.queueUrl,
    MAIN_QUEUE_URL: projectionHandlerQueue.queueUrl,
  },
});

projectionHandlerDLQ.grantReceiveMessages(dlqReprocessorLambda);
projectionHandlerQueue.grantSendMessages(dlqReprocessorLambda);
```

### Automated DLQ Processing (Advanced)

For automated reprocessing after bug fixes:

1. **Monitor DLQ size**
2. **When DLQ has messages**: Check if recent code deployment fixed the issue
3. **Republish messages**: Move messages from DLQ back to main queue
4. **Monitor**: Ensure messages process successfully

**Warning**: Only reprocess DLQ messages after confirming the root cause is fixed. Otherwise, messages will fail again and return to DLQ.

## Alternative: Lambda DLQ (Not Recommended)

### Why Not Use Lambda DLQ?

**Lambda DLQ Configuration**:
```typescript
const projectionLambda = new lambda.Function(this, 'Lambda', {
  // ...
  deadLetterQueue: projectionHandlerDLQ, // Lambda DLQ
});

// EventBridge Rule targets Lambda directly
new events.Rule(this, 'Rule', {
  targets: [new targets.LambdaFunction(projectionLambda, {
    retryAttempts: 10, // EventBridge retries
  })],
});
```

**Limitations**:
- **Less control**: EventBridge retry logic is less flexible than SQS
- **No message persistence**: Failed events may be lost if EventBridge retries are exhausted
- **Harder to monitor**: EventBridge metrics are less detailed than SQS metrics
- **Manual reprocessing**: Harder to republish failed events from Lambda DLQ

**When to Use**: Only if you have very simple retry needs and don't need DLQ message inspection.

## Migration from Direct EventBridge → Lambda

### Steps to Migrate Existing Projection Listeners

1. **Create SQS queue and DLQ** (CDK)
2. **Update EventBridge Rule** to target SQS instead of Lambda
3. **Add SQS event source** to Lambda function
4. **Update Lambda handler** to process SQS events (extract EventBridge event from SQS message body)
5. **Deploy and test**
6. **Monitor** queue depth and DLQ size

### Backward Compatibility

During migration, you can temporarily have both:
- Old rule: EventBridge → Lambda (direct)
- New rule: EventBridge → SQS → Lambda

Then remove the old rule after verification.

## Cost Considerations

**SQS Costs** (AWS Pricing):
- First 1M requests/month: Free
- Additional requests: $0.40 per 1M requests
- Messages stored: $0.0000004 per 1M messages

**Example** (Development workload):
- 10,000 events/month
- ~30,000 SQS requests/month (long polling reduces requests)
- Cost: ~$0.01/month (effectively free)

**DLQ Cost**: Same as SQS (negligible for development)

## Best Practices

1. **Set appropriate visibility timeout**: Lambda timeout + buffer (e.g., Lambda 25s → SQS 30s)
2. **Use long polling**: `receiveMessageWaitTime: 20s` reduces API calls
3. **Batch size**: Start with `batchSize: 1` for error isolation, increase if needed
4. **Monitor DLQ**: Set up alarms for any messages in DLQ
5. **Log comprehensively**: Include event details in error logs for debugging
6. **Idempotency**: Always check if event already processed
7. **DLQ retention**: 14 days is reasonable for investigation (adjust based on needs)

## Summary

**Recommended Pattern**: **EventBridge → SQS → Lambda** with SQS Dead Letter Queue

**Key Benefits**:
- ✅ **10 retries** via SQS `maxReceiveCount`
- ✅ **Automatic DLQ routing** after retries exhausted
- ✅ **Message persistence** and durability
- ✅ **Better monitoring** with SQS metrics
- ✅ **Flexible reprocessing** from DLQ

**Implementation Steps**:
1. Create SQS DLQ
2. Create SQS main queue with DLQ redrive policy (`maxReceiveCount: 10`)
3. Update EventBridge Rule to target SQS (instead of Lambda)
4. Add SQS event source to Lambda
5. Update Lambda handler to process SQS events
6. Add CloudWatch alarms for DLQ monitoring
